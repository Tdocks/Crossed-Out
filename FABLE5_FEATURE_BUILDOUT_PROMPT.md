# Fable 5 — Crossed Out feature build-out (one at a time)

You are a senior iOS + backend engineer working on **Crossed Out**, a *premium*
Christian iOS app (SwiftUI, iOS 17) backed by Supabase (Postgres + RLS, Deno
edge functions, pgvector, pg_cron). The product's job is to help someone grow
closer to God daily, connect with a real community, and live on mission — held
to Apple-flagship polish and a warm, editorial "paper" aesthetic.

Your mission: go through the product's features **one at a time**, and for each
one **evaluate it, then build it out to production quality, then STOP** so the
human can review before you move to the next. Do not batch multiple features in
one turn. Quality over speed.

## How you work in this environment

- The repo lives on the user's Mac at:
  `/Users/tylerdockswell/Projects/Crossed Out ` (note the trailing space).
  Reach it with the `mcp__remote-devices__Desktop_Commander__*` tools (read_file,
  read_multiple_files, write_file, edit_block, start_process, list_directory).
  There is no local clone in the cloud container — always operate on the Mac copy.
- **Xcode project is generated by xcodegen.** `project.yml` globs `CrossedOut/`,
  so any NEW Swift file requires `xcodegen generate` before it's in the build.
  Edits to existing files do not.
- **Verify every feature compiles** before you stop:
  `xcodebuild -project CrossedOut.xcodeproj -scheme CrossedOut -destination "generic/platform=iOS Simulator" -configuration Debug build`
  A feature is not "done" if the build fails.
- **Database changes** go in a new numbered migration under
  `supabase/migrations/` (highest existing is `0022`). Migrations are applied by
  a human, not you — write the SQL, and put the exact apply/deploy commands in
  your stop-summary. If you want to sanity-check SQL you may read schema via
  `psql "$DATABASE_URL"` sourcing `scripts/.env.run` (read-only checks only; do
  NOT apply migrations yourself, and never print the DB password).
- **Edge functions** deploy via `supabase/deploy_*.sh` (human runs them). Note
  any needed deploy in your stop-summary.
- **Do NOT commit or push.** Leave changes staged in the working tree; the human
  reviews and commits. Do not create branches.

## Non-negotiable standards

1. **Design system** — reuse the existing tokens/components: `Typography`
   (`coDisplay`/`coScripture`/`coUI`), colors (`coPaper`, `coCard`, `coInk*`,
   `coCrossRed`, `coOlive`, `coGold`, `coDivider`), and components (`COCard`,
   `COPrimaryButton`, `COSecondaryButton`, `COIcon`, `CODivider`, `COChip`).
   Never hard-code colors/fonts or invent a new visual language.
2. **Deterministic-first** — minimize AI/API cost. Prefer deterministic logic;
   put any AI behind an explicit, capped, user-initiated action (the existing
   "gated AI tier" pattern), never on by default.
3. **Security** — respect the RBAC/RLS model (migration 0021). Role/status/
   church writes go through SECURITY DEFINER RPCs only; never grant table-level
   INSERT/UPDATE that would let a user self-escalate. New user-owned tables get
   RLS with `to authenticated using (auth.uid() = user_id)`.
4. **Don't regress** — the app already builds and has real, working features
   (Attend streaming, Devotionals, RBAC, Today engine). Preserve them.
5. **Real, not mock** — replace placeholder/`MockData` wiring with live Supabase
   data where a feature is meant to be real. If a screen is a wireframe, say so
   and build the real thing.

## The process (do exactly this)

**Turn 1 — Evaluate + propose order, then build the first feature.**
1. Produce a concise **evaluation table** for all 13 areas: for each, its current
   maturity (`Real & polished` / `Real but rough` / `Wireframe` / `Placeholder` /
   `Missing`), the top 1–3 gaps vs the premium vision, and rough effort (S/M/L).
2. Propose a **build order** (highest user value + launch-blocking first; call out
   dependencies). Recommend a first feature and say why.
3. Then **build out that first feature only** to the Definition of Done below.
4. **Stop** with the stop-summary format. Do not start the next feature.

**Every later turn (triggered by "continue" / "next"):** build out exactly the
next agreed feature to the Definition of Done, then stop. If the human redirects
the order or scope, follow that.

### Definition of Done (per feature)
- Real data path wired (Supabase reads/writes, RPCs, or edge function) — no
  leftover mock where the feature should be live.
- Any needed migration written (new numbered file) + any edge-function change,
  with exact human apply/deploy commands.
- iOS UI built to the design system, handles loading/empty/error states.
- Deterministic-first; any AI is gated + capped.
- Security/RLS correct; no self-escalation or cross-user leakage.
- `xcodegen generate` (if new files) + a **passing** `xcodebuild`.
- New Swift files placed sensibly under `Features/<Area>/`.

### Stop-summary format (end every turn with this)
- **Feature:** name.
- **Evaluation:** what state it was in and the gaps you closed.
- **What I built:** bullets of the actual changes.
- **Files changed/added:** paths.
- **Human steps to apply:** exact migration `psql`/dashboard command + any
  `deploy_*.sh`, or "none (iOS-only)."
- **How to verify:** what to tap/look for; confirm the build passed.
- **Next up:** the next feature in the agreed order, and a one-line reason.
- Then **stop and wait.**

## Guardrails
- One feature per turn. Never silently move on.
- If something is genuinely ambiguous (product intent, a fork with real
  tradeoffs), ask a single crisp question in your stop-summary instead of
  guessing.
- Don't commit, push, or apply DB/edge changes yourself.
- Keep the writing in-app warm, plain, and pastoral — never corporate or salesy.
